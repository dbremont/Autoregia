01Overview

AOOS organizes the agent's operative action. It lifts the action-bearing states recorded elsewhere (in the PRS) into a steerable, dependency-aware, capacity-checked, and calendar-coordinated execution structure — and keeps that structure synchronized with the platforms you actually operate through.

Within the Autoregia Personal Viable System Model (PVSM), AOOS is the Operations System (VSM System 1 – Operations): the layer that does things. By making action explicit, ordered, and accountable, it converts an implicit, scattered, anxiety-prone work life into an intelligible, navigable, and steerable operative structure.

i

What AOOS is not. AOOS does not own the underlying record store — that role belongs to the PRS. AOOS operates over PRS records plus a thin operational extension: dependencies, effort estimates, and scheduling state.

02The three components

AOOS is a single System 1 composite of three cooperating components. The conceptual separation between them is real; the deployment boundary is one project.

A
“What to do”

Work Organization & Registration

The action hierarchy — action constructs (tasks, projects, routines, commitments, initiatives, objectives) plus their dependency graph, effort estimates, and logical ordering. Operates over PRS records plus a thin operational extension.

B
“When, against the real calendar”

Work Calendarization

Takes the logical action structure and resolves it against a concrete timeline: recurrence expansion, deadline projection, conflict detection, workload-vs-capacity checks, and policy gating. Block lifecycle: tentative → confirmed → completed.

C
“How it connects”

Platforms Integration

The boundary with the outside world. Adapters normalize external systems into AOOS terms. Google Calendar is the primary platform (OAuth 2.0, two-way sync); future platforms plug in via the adapter interface.

Components [B] and [C] together realize what the original formulation called Work Calendarization: [C] brings the platform's calendar in; [B] reasons over it.

03Getting started

Run the prototype

AOOS ships as a Flask mock-API server plus a vanilla HTML/CSS/JS client. It runs in mock mode by default — fully functional on local data, no credentials required.

# from the repository root
python3 -m venv env && source env/bin/activate     # once, shared venv
pip install -r aoos/requirements.txt
python3 aoos/server.py

# open http://localhost:5005\\
  
The server listens on port 5005. Override it with the AOOS_PORT environment variable. Seed data lives in aoos/data/mock_actions.json and mock_blocks.json; regenerate it with python3 aoos/data/gen_mock.py.


  
First steps

  
    
    
  1. Open the Dashboard for an at-a-glance summary of your action inventory and schedule.
    2. 
    
  2. Go to Actions and press N (or New Action) to create your first action construct — set its kind, effort, and scheduling state.
    3. 
    
  3. Open an action's detail to add dependencies and schedule a calendar block.
    4. 
    
  4. Visit the Calendar to see your blocks across day / week / month / year views; conflicts are surfaced automatically.
    5. 
    
  5. Connect Google Calendar when you want live two-way sync (see setup).
    6. 
  


  
04Core concepts


  
Action constructs

  
An action construct is the structured unit of work the agent is (or may be) performing. AOOS recognizes six kinds, forming a logical hierarchy:

  

    
KindRole

      
ObjectiveLong-horizon outcome; the top of a hierarchy.

      
InitiativeA coherent program that advances one or more objectives.

      
ProjectA bounded body of work producing a deliverable.

      
TaskA discrete, actionable unit of work.

      
RoutineA recurring, repeatable activity.

      
CommitmentA promise or obligation to an external party.

    

  

  
Each construct carries scheduling state (unscheduled, scheduled, deferred, in-progress, done), an optional effort estimate (hours, points, or sessions, with a confidence), and a capacity profile (time, focus, energy, attention; deep / shallow / low).


  
Dependencies

  
Constructs are connected by typed edges forming a dependency graph. Common edge kinds:

  

    
Edge kindMeaning

      
part-of / containsComposition / hierarchy membership.

      
depends-on / blocked-byOrdering: cannot start until the target resolves.

      
enables / required-byEnabling / requirement relations.

      
implements-objectiveLinks a project/initiative to an objective.

      
fulfills-commitmentLinks work to an obligation.

      
uses-capabilityReferences a PTOCS capability.

      
governed-byReferences a PPS policy.

      
recurs-from / spawned-fromLinks a routine instance to its origin.

    

  

  
AOOS computes the critical path, surfaces blockers, and offers a next-action surface (unblocked, in-horizon, capacity-available constructs).


  
Calendar blocks

  
A calendar block is the concrete temporal placement of work: a start, an end, a status, and an optional link to a construct or external event. Blocks have a lifecycle — tentative → confirmed → completed (or cancelled) — and carry conflict flags when they overlap, overload a period, or violate a policy.

The interface follows the Autoregia UI specification — a warm-parchment Oxford aesthetic, calm and consistent, with shared design tokens, the Spectral / Inter / IBM Plex Mono type system, and self-hosted Lucide icons.

Command palette

  
Press Ctrl/⌘ + K to open the universal command palette. Type to jump to any view, run a command (new action, sync, export, open help, open these docs), or find an action by its record id or kind.

Press F1 anywhere for a compact overlay of shortcuts, views, and the three components — the short-form companion to this document. It links here for the full reference.


  
✦

06Data model

  
Records live in the PRS (single source of truth). AOOS adds a thin operational extension persisted as JSON + SQLite: action constructs, dependency edges, calendar blocks, external mappings, and sync state. The authoritative shapes are defined in spec/aoos/schema.json; the export endpoint returns data conforming to it.

  
// action construct (abridged)
{
  "id": "act_...",
  "record_id": "...",          // PRS record id (source of truth)
  "kind": "Task",
  "scheduling_state": "scheduled",
  "pinned": false,
  "strategic": { "project": "...", "objective": "..." },
  "effort": { "value": 3, "unit": "hours", "confidence": "Medium" },
  "capacity": { "resource": "focus", "band": "deep" },
  "dependencies": [ { "kind": "blocked-by", "target": "act_..." } ]
}

// calendar block (abridged)
{
  "id": "blk_...",
  "title": "...",
  "starts_at": "2026-06-29T09:00:00Z",
  "ends_at":   "2026-06-29T11:00:00Z",
  "status": "confirmed",      // tentative | confirmed | completed | cancelled
  "action_id": "act_...",
  "external_id": "...",        // set when mirrored from Google Calendar
  "sync_state": "clean",       // clean | dirty | conflict
  "conflict_flags": []
}

The Flask backend exposes the three components as a REST API. All endpoints are JSON; times are ISO-8601 (UTC, Z suffix).

GET /api/actionsAList actions (filter: kind, scheduling_state, project, objective, pinned, q).

      
POST /api/actionsACreate an action construct.

      
GET/PUT/DELETE /api/actions/<id>ARetrieve / update / delete an action.

      
POST /api/actions/<id>/pinAToggle pin.

      
GET /api/hierarchyAObjective → Initiative → Project → Task tree.

      
GET /api/blocksBList blocks (filterable).

      
POST /api/blocksBCreate a block (with automatic conflict detection).

      
GET/PUT/DELETE /api/blocks/<id>BRetrieve / update / delete a block.

      
GET /api/calendar/day?start=BBlocks for a single day.

08Google Calendar setup

  
The prototype runs in mock mode by default — fully functional on local block data, no credentials required. To enable live two-way sync:

  
    
    
  1. Create a Google Cloud project; enable the Google Calendar API.
    2. 
    
  2. Create an OAuth 2.0 Client ID (application type: Desktop app).
    3. 
    
  3. Download the client secret as config/client_secret.json inside aoos/ (or set AOOS_GC_CLIENT_SECRET to its absolute path). Never commit it.
    4. 
    
  4. Install the Google libraries (already in requirements.txt): google-auth-oauthlib google-api-python-client.
    5. 
    
  5. Open the Google Calendar view and click Connect; complete the consent flow. The refresh token is stored in config/token.json.
    6. 
    
  6. Click Sync (header) or Sync Now to pull remote events and push local dirty blocks.
    7. 
  

  
Scope: https://www.googleapis.com/auth/calendar\ (read/write) for two-way sync. The header pill (GC: …) shows the live connection state at a glance.


  
!
Security. OAuth credentials are not part of this repository. The agent provides them out-of-band. Tokens are stored locally only; the config/ directory is gitignored.


  
09Import & export

  
Use Export to download the full dataset (actions + blocks) as JSON conforming to spec/aoos/schema.json. Use Import to merge a file back in — entries are matched by id (merge-by-id), so existing records are never duplicated or overwritten by import.


  
10Troubleshooting

  

    
SymptomResolution

      
GC: mock in the headerExpected in mock mode. Add client_secret.json and connect from the Google view to go live.

      
Sync reports conflictsOpen the offending block in the calendar; the detail view lists conflict flags (overlap, overload, policy violation).

      
Port already in useSet AOOS_PORT to a free port and reopen http://localhost:\<\;port\>\;\.\\
      
Fonts / icons missingEnsure you're serving via the Flask server (which mounts static/), not opening the HTML file directly.

      
Help modal won't openMake sure focus is not in a text field; ? is suppressed while typing (use F1 instead).
